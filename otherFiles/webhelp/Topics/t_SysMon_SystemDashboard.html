System Dashboard

The System Dashboard provides general information as well as information related to the JVM and the host system of the currently selected VersaLex. The System Dashboard is displayed.

General Information Table

The General Information table provides information identifying the product, serial number, alias, version, version available, pool name, IP address, and date/time the application was last started.

There are no sub-reports from this library component.

CPU Usage Percentage Chart

The CPU Usage Percentage chart displays the current CPU load as a percentage of total CPU capacity related to the VersaLex server.

A continuously high CPU load can indicate that another VersaLex should be added to the system to spread out the load.

Access the CPU History sub-report by clicking on the needle in the graph.

CPU History Report

The CPU History report displays the CPU load as a percentage of total CPU capacity over time. By default, it will show the last 60 minutes of history. Change the Start date/time of the chart and the number of Minutes displayed using the parameter control box. Use the zoom slider at the bottom to zoom in closer to a specific time. Hovering the cursor over a specific data point on the chart will display information about that data point.

System Uptime Percentage Chart

The System Uptime Percentage chart provides the percentage of time VersaLex has been running. It measures the availability of VersaLex and its installed components.

There are no sub-reports from this library component.

Heap Memory Chart

The Heap Memory chart displays the percentage of heap memory used. Heap memory is the portion of memory in which dynamically allocated memory resides. Memory allocated from the heap will remain allocated until either the memory is freed up or the program terminates. If all references to allocated memory are lost you have a memory leak: the memory has still been allocated, but there is no easy way of accessing it anymore. Leaked memory cannot be reclaimed for future memory allocations, but when the program ends the memory will be freed up by the operating system.

The maximum heap memory can automatically be determined by the JVM, or it can be configured as a specific maximum. If the system runs out of heap memory, it could be a memory leak as mentioned above, but it could also be that the maximum value is set too low for the amount of VersaLex traffic. If the maximum is already set high and VersaLex is still running near the maximum, that indicates that another VersaLex should be added to the system so the load can be spread out between more VersaLexes.

Access the Heap Memory History sub-report by clicking on the bar.

Perm Gen Memory Chart

The Perm Gen Memory chart displays the percentage of perm gen memory used.

Perm Gen Memory is a heap in the JVM dedicated to storing the JVM's internal representation of Java Classes and interned String instances. In the Oracle JVM, the heap is a fixed size. If an attempt is made to load more class instances than will fit in this area, an Out Of Memory Error occurs, even when there is still plenty of space on the other heaps.

A problem in the Perm Gen Memory area will typically show a configuration problem. For instance, if JMS URI is being used, the classes of the specific JMS provider are also loaded into the VersaLex JVM. This can be an issue with the Perm Gen Memory.

Access the Perm Gen Memory History sub-report by clicking on the bar.

Overall Transfer Rate Chart

The Overall Transfer Rate chart displays the overall inbound/outbound transfer rates.

Overall Transfer Rate monitors the speed of inbound and outbound transfers in which files are currently being executed. This is an accumulation of all transfers currently being executed, and can be reduced by the trading partner(s) if their connectivity is lower than the maximum bandwidth of the VersaLex.

This can be used to see if the bandwidth is reaching a system maximum, at which time another VersaLex should be added to spread out the traffic.

Access the Overall Transfer Rate History sub-report either by clicking on the bars or the bar label.

Thread Count Chart

The Thread Count chart displays a count of the active and deadlocked threads. It also shows the peak number of threads since the last service restart.

Threads are similar to processes. Threads, like processes, are time-sliced by the kernel. On uniprocessor systems, the kernel uses time slicing to simulate simultaneous execution of threads in much the same way it uses time slicing with processes. On multiprocessor systems, threads are able to run simultaneously, just like two or more processes can.

This can be used to monitor the typical number of threads for their system. This will vary depending on the customer's outbound and inbound load. If the customer sees a significantly higher or lower number of threads during normal traffic, this could indicate an issue that something is not functioning correctly in the system.

Access the Thread Count History sub-report by clicking one of the bars.

Thread Count History Report

The Thread Count History report displays the Active, Peak, and Deadlock thread counts over time. By default, it will show the last 60 minutes of history. Change the Start date/time of the chart and the number of Minutes displayed using the parameter control box. Use the zoom slider at the bottom to zoom in closer to a specific time. Hovering the cursor over a specific data point on the chart will display information about that data point.

System Queues Chart

The Thread Count chart displays the percentage of each of the system queues in use.

This measures the size of the internal queues that are being used in memory and include Transfer, System, and Operator Audit Trail queues. If any of these queues are full, it will affect the performance of VersaLex.

Access the System Queue Percentage Used History sub-report by clicking the bars or the bar labels.

System Queue Percentage Used History Report

The System Queue Percentage Used History report displays the percent of the system queues in use over time. By default, it will show the last 60 minutes of history. Limit the history displayed to one or more queues by selecting them in the Queue filter. Change the Start date/time of the chart and the number of Minutes displayed using the parameter control box. Use the zoom slider at the bottom to zoom in closer to a specific time. Hovering the cursor over a specific data point on the chart will display information about that data point.
